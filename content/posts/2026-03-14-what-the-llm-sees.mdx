The model sees exactly what you send it. No persistent memory, no background awareness of prior sessions. What you call a "conversation" is a text structure your app assembles and re-uploads on every request.

Two patterns: (a) stateless — your app serialises the full exchange into a messages array and re-sends it each turn; (b) stateful — the provider holds history server-side and you continue from a stored reference. Both are in production use. The tradeoff is control vs. convenience.

The messages array is the foundation either way. Even stateful APIs manage an array under the hood. Understanding it is what lets you reason about costs, failure modes, and the limits of context management.

---

## The messages array

Every LLM API call that involves conversation takes a `messages` parameter — an ordered array of objects. Each object has a `role` and a `content`. The model reads the array top-to-bottom, infers the conversation so far, and continues it.

```go
resp, err := client.Messages.New(ctx, anthropic.MessageNewParams{
    Model:     anthropic.F(anthropic.ModelClaudeOpus4_6),
    MaxTokens: anthropic.F[int64](1024),
    System: anthropic.F([]anthropic.TextBlockParam{{
        Type: anthropic.F(anthropic.TextBlockParamTypeText),
        Text: anthropic.F("You are a helpful assistant."),
    }}),
    Messages: anthropic.F([]anthropic.MessageParam{
        anthropic.NewUserMessage(anthropic.NewTextBlock("What is a context window?")),
        anthropic.NewAssistantMessage(anthropic.NewTextBlock("A context window is the maximum amount of text...")),
        anthropic.NewUserMessage(anthropic.NewTextBlock("How does it affect cost?")),
        // ↑ the model sees ALL of this, not just the last message
    }),
})
```

## Watching the context window grow

The context window **grows with every turn**. Not by the size of the new message — by the cumulative size of every message ever sent in the session.

<DynamicComponent id="agent-memory-v2" />

Watch the right panel. Every send extends the array — turn 3 ships turns 1, 2, and 3. By turn 10 you're sending 10× the data from turn 1. And the model isn't keeping score: each call it wakes up fresh, reads the whole array, reasons, responds. Groundhog Day. Your app is the one remembering; the model is stateless.

Long sessions get expensive fast. "Just keep chatting" is not an architecture.

## The request cycle

What actually happens between you hitting Enter and seeing a response:

```mermaid

    rect rgb(14, 27, 46)
        note over U,App: User turn
        U->>App: types message
    end
    rect rgb(14, 46, 26)
        note over App,DB: History load + append
        App->>DB: load conversation history
        DB-->>App: messages[]
        App->>App: append new user message
    end
    rect rgb(30, 14, 46)
        note over App,API: Full array sent every call ✈
        App->>API: POST /messages (full array)
        API-->>App: assistant response
    end
    rect rgb(46, 26, 14)
        note over App,DB: Store + display
        App->>DB: save assistant message
        App-->>U: display reply
    end
```

In stateless flows, the expensive step is obvious: `POST /messages (full array, every turn)`. Your app owns the history — load, append, re-upload, repeat. In stateful flows, the provider stores context server-side and you continue from a reference; no full re-upload per turn.

Stateless is a feature, not just a limitation. You control exactly what the model sees — redact turns, inject context mid-session, summarise and compress older segments, swap in retrieved documents. Stateful shifts that control to the provider. Which you want depends on what you're building.

## Token costs are cumulative

Every object in the messages array costs tokens — and tokens cost money and latency. This is the part that bites teams hardest in production.

<DynamicComponent id="context-demo-v1" />

The numbers are approximate. The shape is what matters: **token usage grows with every turn**. A 20-turn conversation isn't 20× the cost of 1 turn — it's closer to (1 + 2 + 3 + ... + 20) = 210× the cost of the first message. O(n²) if you're not careful.

The practical limits:

| Model | Context limit | Rough cost signal |
|---|---|---|
| Claude Haiku 4.5 | 200k tokens | Cheapest per token, fast |
| Claude Sonnet 4.6 | 1M tokens | Balanced; long-context premium above 200k input tokens |
| Claude Opus 4.6 | 1M tokens | Most capable; long-context premium above 200k input tokens |
| GPT-5.4 | 1.05M tokens (~1.1M) | Long-context penalty above 272k input tokens; max output is lower |

1M tokens sounds generous until long-context pricing kicks in. The exact trigger is provider/model specific and materially higher than 32k for the major 2026 models above (200k for Claude Sonnet/Opus 4.6 and 272k for GPT-5.4 input pricing penalties). The extended window is there when you need it; it is not uniformly priced. A typical English word is about 1.3 tokens, so 200k tokens is already a long novel. Budget pressure still arrives before the hard limit.

A third option — **RAG (retrieval-augmented generation)** — avoids storing conversation history altogether. Instead of replaying the conversation, you embed each turn and retrieve the semantically relevant ones at query time. This is what production chatbots with millions of users do. It's more infrastructure, but it's the only approach that scales to arbitrarily long sessions.

**Prompt caching** is effective but commonly misunderstood. It reduces compute cost — the provider doesn't re-process stable prefixes — but it does not reduce how much data you send. Your app still ships the full array every request; the provider hashes it, finds the cached KV state for the matched prefix, and skips re-processing it. Anthropic charges ~10% of the original write cost on cache hits. A single token mutation invalidates the prefix. Effective when the stable part is large; not a bandwidth solution.

**Context compaction and stateful APIs** are another path. Provider-managed state can eliminate the reconstruct-and-re-upload loop by letting you continue from stored history. OpenAI's Responses flow can do this when state is stored and you continue from a prior response/thread reference; Anthropic also supports server-side context management features. The tradeoff is control and transparency: depending on product mode, you may have less ability to inspect, surgically redact, or deterministically reconstruct the exact prompt seen by the model. For many chatbots that's acceptable; for agent systems with strict auditability, teams often keep explicit context assembly.

## Tool calls: the hidden cost

Tool use adds a wrinkle that trips up a lot of people. When the model outputs a tool call, generation is finished — not paused. The model isn't waiting for your function to return. It's done. A new request starts from scratch when you send back the result, with the full history plus the tool output appended. Every tool call is a full API round-trip. Every round-trip ships the whole context.

The steps:

1. Model outputs a tool-call request → append as `assistant` message
2. Run the tool
3. Append the result as a `tool` message
4. New API call with the extended array

```go
// What one tool-call cycle looks like in the messages slice
messages := []anthropic.MessageParam{
    anthropic.NewUserMessage(anthropic.NewTextBlock("Search for recent papers on RAG")),
    {
        Role: anthropic.F(anthropic.MessageParamRoleAssistant),
        Content: anthropic.F([]anthropic.ContentBlockParamUnion{
            anthropic.ToolUseBlockParam{
                Type:  anthropic.F(anthropic.ToolUseBlockParamTypeToolUse),
                ID:    anthropic.F("toolu_01"),
                Name:  anthropic.F("web_search"),
                Input: anthropic.F[any](map[string]any{"query": "RAG papers 2025"}),
            },
        }),
    },
    {
        Role: anthropic.F(anthropic.MessageParamRoleTool),
        Content: anthropic.F([]anthropic.ContentBlockParamUnion{
            anthropic.ToolResultBlockParam{
                Type:      anthropic.F(anthropic.ToolResultBlockParamTypeToolResult),
                ToolUseID: anthropic.F("toolu_01"),
                Content:   anthropic.F(`[{"title": "RankRAG...", "year": 2025}]`),
            },
        }),
    },
    anthropic.NewAssistantMessage(anthropic.NewTextBlock("Here are three recent papers on RAG...")),
    // ↑ 4 messages for what looks like 1 exchange
}
```

A single agentic step with 3 tool calls adds 6 messages to the context before the model sends a final reply. In a 10-step agent loop with 3 tool calls per step, you're sending 60+ messages on the final call — many of which are just scaffolding the model generated to reason through intermediate steps.

If your agent is slow or expensive and you can't figure out why, count your messages. The answer is usually there.

One aside: MCP (Model Context Protocol) is standardising the format of tool-call messages across providers. If you're building multi-provider agents, the wire format is converging even if the SDK APIs haven't caught up yet.

## What this means in practice

A few rules that fall out of all this:

**Keep system prompts surgical.** Every token in the system prompt costs on every call. 1,200 tokens of hedging is 3× the cost of 400 tokens that are precise. Write it like you're paying per word — you are.

**Log the full array, not just the last message.** When an agent loop breaks, you need to see exactly what the model was given. The surprising input is almost always buried in an earlier turn.

**Budget by turns, not by session.** Hard limit on how many turns before you force summarisation or reset. Don't let users discover your context limit by accident.

**The model doesn't know what it doesn't see.** Prune a message and the model reasons as if it never happened. Superpower when you're cleaning up bad turns. Footgun when you accidentally hide what it needed.

---

The context window is the most concrete, measurable thing in an LLM application. Everything else — prompt quality, model choice, temperature — is fuzzy. This one is arithmetic. Know what you're sending, or what you're delegating. Track how it grows. Have a plan for when it doesn't fit.
